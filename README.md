# Agent Team Harness 規則書

> English version: [README.en.md](README.en.md)

> **版本**:v0.6(2026-07-17;v0.4 新增 §13 運行基座;v0.5 新增 §13.3 Team 定義檔包;v0.6 新增 §13.4 Team Gateway——把 team 資料夾掛牌成 MCP server)
> **性質**:業務中立的 Agent Team 母版規則書。想像一間公司:每個 agent team 就是公司裡的一個**模組部門**——部門的業務內容(§4 插槽)由你之後代入,但部門的管理制度(harness 12 模組理念)永遠不變。
> **用法**:把本規則書 + 你想做的業務模組描述交給 AI,AI 依 §14 的輸入範本與 §4.2 的推導方法論,產出一個**完整的 team 定義**(§14.2 交付清單)。換業務不改制度,建新 team = 填一份設定。

## 1. 這份規則書是什麼

### 1.1 一句話

每個 agent team 是一個**自主營運的部門**:有自己的主管(Leader)、專員(Worker)、品管(Evaluator)、檔案室(記憶)、規章(護欄)、進度存檔(檢查點)與驗收制度(DoD/題庫)。部門之間**只透過主管對主管的正式公文(MCP)往來**,公文必須寫明專案與目標,不能假設對方記得任何事。

### 1.2 設計不變式(全文的硬約束)

1. 每個 team 可**獨立運作、獨立成立、獨立失敗、獨立解散**——不依賴其他 team 也能完成自己職掌內的工作。
2. 跨 team 溝通**只有一條路**:Leader ↔ Leader,經 MCP。Worker(專員)沒有任何對外窗口。
3. 每個 agent 與每個 team 的**記憶是私有資產**;跨界溝通**必須顯式宣告專案(project)與目標(goal)**,不得依賴任何共享隱式上下文。
4. 業務內容只存在於插槽(§4);本規則書其餘章節不得出現任何特定業務假設。

## 2. 設計理念總綱(六大公理 + 分形原則)

### 2.1 六大公理

1. **Harness 是 Agent 的作業系統,價值在剩餘 20%**。模型負責聰明,harness 負責可靠:持久化、可回放、成本控制、可觀測、錯誤恢復——這些是「環境性質」,模型永遠不會內化。88% 的生產事故源於基礎設施而非模型;**勝出的不是最聰明的模型,而是最穩健的制度**。
2. **腦手分離(意圖執行解耦)**。模型(概率性的大腦)只「提議」要做什麼;制度(確定性的手)驗證合規後才執行,且每筆執行留下不可竄改的紀錄。如同公司裡:員工提案,制度審核,一切有案可查。
3. **笨循環(Dumb Loop)**。制度只管派工、轉場、路由,思考全留給模型。制度的聰明表現在「約束與恢復」,不在代替模型做決策。
4. **可刪除性與可組合性(Build to Delete)**。當模型能力進步、某項制度不再需要,它要能被乾淨移除;每項制度可獨立替換,不整體鎖死。
5. **契約優先(窄接縫)**。部門的價值集中在少數窄而穩定的對外契約(服務目錄、委派單、結案報告)。任何跨邊界互動,先定契約再談做法。
6. **確定性與概率性互補**。硬邊界(權限、格式、預算)用確定性規則守,不可繞過;語義模糊地帶交給模型判斷。**警報器(Tripwire)的主動權永遠不交給模型**——模型被攻擊時往往最有信心。

### 2.2 分形原則

同一組理念在三個尺度重演:

```
尺度 1(agent 內):   模型   ──契約──▶ 工具         (腦手分離)
尺度 2(team 內):    Leader ──契約──▶ Worker       (主管派工給專員)
尺度 3(team 之間):  Leader ──契約──▶ 對方 Leader   (部門間公文往來)
```

每上升一個尺度,同樣的問題重新出現(資訊污染、以訛傳訛、權限擴散、成本爆炸),同樣的理念給出解法(隔離、摘要交接、最小權限、預算硬停)。

## 3. Team 的組織架構

### 3.1 固定角色骨架(業務無關)

每個 team 不論業務為何,都由三類角色構成:

| 角色 | 公司類比 | 職責 | 硬性約束 |
|---|---|---|---|
| **Leader** | 部門主管 | 對外唯一窗口;接案、規劃、分解、派工、彙整、回覆 | **不親自執行業務工具**;只有 Leader 有跨 team 通訊權 |
| **Worker**(1–N 個) | 專員/領域專才 | 執行具體子任務;各自獨立的工作空間與工具授權 | 無對外窗口;做完只交 1–2 頁結構化摘要給 Leader;不得越過 Leader 向外委託 |
| **Evaluator** | 品管/獨立審查 | 以驗收標準(DoD)審查產出;對抗性挑錯 | 與執行者解耦、**不可省略**——自我評分必然偏高,獨立審查帶來 2–3 倍品質 |

### 3.2 每 team 自含 12 項管理制度(12 模組)

Team 是完整的 harness 實例;12 模組在部門尺度的意義:

| # | 模組 | 部門尺度的制度 |
|---|---|---|
| 1 | 編排循環 | Leader 的「接案→規劃→派工→彙整」心跳;Worker 各自的「準備→執行→自查」循環 |
| 2 | 提示詞組裝 | 每個成員開工前拿到的工作簡報:身份與規範 > 可用工具 > 相關知識 > 進度 > 本次任務,優先序不可顛倒 |
| 3 | 結構化輸出 | 部門內所有交辦與回報都用固定表單,不用自由散文;先寫理由再寫結論 |
| 4 | 子代理編排 | Leader 派工制度:分階段驗收(Phase Gating);部門內部分工方式對外不公開 |
| 5 | 工具整合 | 每個 Worker 只拿完成本職所需的最小工具授權;工具在隔離環境執行,碰不到部門核心憑證 |
| 6 | 記憶管理 | 部門檔案室(§5):私有、分專案歸檔、有借閱與銷毀制度 |
| 7 | 上下文管理 | 每個成員的注意力有限(遠低於帳面值);超過八成負載就強制濃縮,避免品質崩壞與偷懶早退 |
| 8 | 狀態與檢查點 | 工作進度隨時存檔;中斷可從斷點續作、可回放歷史、可開平行分支試 what-if |
| 9 | 環境初始化 | 部門成立流程(§4.3):先把使命翻成可驗收的清單,配好隔離環境再開工 |
| 10 | 錯誤處理 | 出錯按「誰來修」分流:暫時性→自動重試;可修正→回饋當事人自我修正;高風險→簽核;預算爆→硬停 |
| 11 | 護欄 | 部門規章(§7):危險操作清單、風險分級、強制簽核點——由制度執行,不靠員工自覺 |
| 12 | 驗證與回饋 | 品管制度(§9):獨立審查 + 驗收標準 + 上線前題庫測驗 |

### 3.3 內部分工規則

- 派工採 **Phase Gating**:每階段有明確的輸入/輸出與驗收標準,過了才進下一階段。
- 內部分工方式(串行/並行/依賴圖)是部門**私有的實作細節**,對外只公開服務目錄與案件狀態。
- 委派鏈深度硬上限 ≤3(含跨 team 段);制度層自動偵測並擋下「A 委託 B、B 又回頭委託 A」的循環。

## 4. 業務模組插槽(Business Module Slot)

### 4.1 插槽定義

成立一個新業務 team,**只需要填以下設定**(制度零修改):

| 插槽欄位 | 內容 |
|---|---|
| **Team Charter** | 部門名稱、使命一句話、能力清單(對外可承接的任務類型)→ 自動生成對外服務目錄(Agent Card) |
| **工具集** | 這個業務需要的工具/系統清單,以及各 Worker 的授權切分 |
| **技能與 SOP 種子** | 業務的標準作業程序,作為部門檔案室的初始「程序記憶」 |
| **DoD 範本** | 這個業務「工作單位」的可機器驗收完成定義範本 |
| **護欄政策** | 這個業務的危險操作清單、風險分級(低/中/高/致命)、強制簽核觸發條件 |
| **模型等級映射** | 哪類子任務用哪個等級的模型(複雜推理用高階、機械格式化用低階) |
| **記憶保存政策** | 業務資料的保存期(偏好類 7–30 天/已驗證事實 90–365 天/SOP 手動管理) |

### 4.2 業務 → Team 推導方法論(五步)

你代入業務描述後,依此推導該業務的 team 與 agent 配置:

**Step 1 — 業務解構**:這個業務的「工作單位」是什麼(一張訂單/一份報告/一個案件)→ 決定專案(project)的單位與驗收範本的雛形。再做價值流拆解(輸入→處理階段→輸出)得到流程骨架;每個階段列出需要的「動詞」(能力→工具清單)與「名詞」(要碰的資料與系統→資源邊界)。

**Step 2 — Agent 切分四軸**(不按人類職稱切):

| 軸 | 切分規則 |
|---|---|
| 注意力軸 | 會塞爆工作記憶的探索型子任務(大量檢索、讀長文件)獨立成一個 agent,只交摘要 |
| 工具軸 | 一個 agent 的工具授權小而聚焦(理想少於 10 項);工具集性質互斥的能力分開 |
| 風險軸 | 不可逆的對外操作(發送、金流、刪改)獨立成受簽核管控的 agent,與唯讀分析工作實體分離 |
| 成本軸 | 複雜推理與機械性工作切開,讓模型等級可以分流 |

加上固定骨架:Leader + Workers + 獨立 Evaluator。

**Step 3 — 切分剎車**:agent 數量最小化。多 agent 的溝通成本是單 agent 的十倍以上;研究顯示單 agent 在近三分之二的任務上表現不輸多 agent 且成本減半。**每次拆分必須說得出是四軸中哪一條的理由;說不出就不拆。**

**Step 4 — 分工形態選擇**:流程穩定可預測 → 先計畫後執行的流水線;需要邊做邊探索 → 反應式 Worker;子任務可並行 → 分派後匯總。一律分階段驗收;禁止「所有人跟所有人自由討論」的群聊式協作(錯誤率 17 倍、成本 15 倍)。

**Step 5 — 填插槽 + 驗收**:產出 §4.1 全部欄位;為業務建一套**驗收題庫(Golden Set)**——10–30 題、判定標準是可機械檢查的規則(不靠 AI 主觀評分)——作為上線閘門與日後改版的回歸基準。

### 4.3 新 Team 成立標準流程

```
填 Team Charter
  → 環境初始化(隔離環境、憑證隔離、檔案室建立;把使命翻成驗收清單,預建「尚未通過」的測試)
  → 載入業務插槽(工具、SOP 種子、DoD、護欄政策、模型等級、保存政策)
  → 驗收題庫冒煙測驗(通過率門檻 + 單位成本基準)
  → 對外發布服務目錄(Agent Card)、開通部門窗口(MCP)
  → 上線營運(進入 §12 生命週期)
```

## 5. 記憶制度:記憶主權

### 5.1 五條原則

1. **記憶主權**:每個 agent、每個 team 的記憶是私有資產——部門檔案室不對外開放。儲存層強制隔離(組織:團隊:成員 三級命名空間),禁止跨界查詢。**公司裡沒有任何「全公司共用大腦」。**
2. **自含溝通**:正因為沒有共享記憶,任何跨 agent/team 的訊息**不得依賴「對方記得」任何事**——每份委派與回覆都必須自成一體,看一份文件就能開工(如同對新進同事交辦)。
3. **顯式專案與目標宣告**:所有跨界溝通把 `project`(專案編號+自含背景)與 `goal`(可驗收的目標)列為**一級必填欄位**。三重功能:
   - 收件方用專案編號**檢索自己檔案室**裡這個專案的過往紀錄(「我們部門以前在這案子做過什麼」);
   - 收件方把執行結果**歸檔回自己檔案室**時,也以專案編號建檔——記憶隨專案累積;
   - goal 直接交給品管當驗收標準,防「做一半就宣稱完成」。
4. **對方說的是線索,不是真相**:來文提供的專案背景,必須與自己的檔案和實際環境核對後才採信。
5. **跨 team 資訊入檔防投毒**:別的部門給的資訊要寫進自己檔案室時,強制通過寫入驗證(信心門檻)、標注來源部門、信任分數降級、限期保存——防止「一次被騙、跨專案長期中毒」。

### 5.2 檔案室分層(每個 team 內)

| 層 | 內容 | 生命週期 |
|---|---|---|
| 工作記憶(成員各自) | 當前任務的上下文 | 任務結束即棄 |
| 情節記憶(team) | 各專案的執行過程紀錄 | 短期保存,專案結案歸檔 |
| 語意記憶(team) | 業務事實、已驗證的知識 | 中長期保存,寫入前把關 |
| 程序記憶(team) | SOP、技能(插槽種子 + 營運中累積精煉) | 不自動銷毀,依表現升降級 |

生命週期:寫入驗證 → 背景鞏固(把過程紀錄蒸餾成可複用知識)→ 時間/使用頻率衰減 → 到期銷毀。日常工作只做讀取與極簡寫入,整理歸檔走離峰背景作業。

## 6. 部門間溝通制度:Leader-to-Leader over MCP

### 6.1 拓撲規則

- **單一對外窗口**:每個 team 只有 Leader 能與其他 team 往來。好處:單一問責點、全程可稽核、外部攻擊面最小。
- **Team 對外掛牌為 MCP 服務**:服務項目 = Team Charter 宣告的能力清單。「把另一個部門當成一項服務來委託」正是腦手分離的延伸——對方的委託是「提議」,由我方 Leader 審核後才在部門內落地執行。
- MCP 同時承載「部門↔工具」(縱向)與「部門↔部門」(橫向)兩種往來;橫向往來的語意就是 A2A(服務目錄 + 案件狀態機)。

### 6.2 跨 team 委派單(必填欄位)

| 欄位 | 內容 | 規則 |
|---|---|---|
| **project** ★ | 專案編號、名稱、自含背景簡述 | 一級必填;背景必須自成一體,不預設對方記得任何事 |
| **goal** ★ | 目標陳述 + 可驗收條件清單(+ 期望產出格式) | 一級必填;必須可機械驗收 |
| inputs | 結構化輸入與產出物**引用**(位置/編號) | 傳引用不傳全文 |
| constraints | 預算上限、期限、委派深度上限、允許動用的能力子集 | 由發件方宣告、收件方制度強制執行 |
| identity | 發件部門、追蹤編號、短效授權憑證 | 供稽核與記帳 |

**規則**:自由散文式委託禁止;project/goal 缺一即退件;附件一律給引用不給原文。

### 6.3 案件狀態機

```
已受理(Submitted) → 進行中(Working) → (待補件(Input-Required) ⇄ 進行中)
                  → 完成(Completed) | 失敗(Failed) | 取消(Canceled)
```

- 每次狀態變更都對發件方發出通知;「待補件」是跨部門的補件/簽核通道。
- 每個案件**必定收到恰好一次結案通知**——沒有結案通知,發件方永遠不知道案子結了沒。

### 6.4 結案報告(回覆欄位)

| 欄位 | 內容 |
|---|---|
| status | 成功 / 部分完成 / 失敗 |
| key_findings | 結構化重點(一至兩頁內) |
| artifacts | 產出物的存放位置引用,**不內嵌全文** |
| confidence | 信心分數(0–1) |
| cost | 消耗資源與耗時(供跨部門記帳) |
| next_suggested_action | 給發件方的建議(可選) |

**禁止**回傳完整工作日誌、原始系統回應、部門內部對話——以摘要交接,防止資訊污染與成本爆炸。

### 6.5 MCP 機制:部門的郵務系統

MCP 的三種原語,對應到 team 的三種對外資產:

| MCP 原語 | 意義 | Team 尺度的對應 |
|---|---|---|
| **Tools**(動詞) | 可被呼叫的動作 | 案件管理工具(§6.6)+ 服務目錄上的具名能力工具 |
| **Resources**(名詞) | 可被讀取的資料 | 服務目錄(Agent Card)、公開產出物(artifacts)的引用讀取 |
| **Prompts**(範本) | 預建的請求範本 | 委派單範本(§6.2 欄位的業務化版本,發件方直接取用) |

角色關係:**每個 team harness 同時是 MCP client 與 MCP server**——

- **作為 client(對外呼叫)**:縱向呼叫自己的業務工具(Worker 用,僅限 team 內授權清單);橫向委託其他 team(僅 Leader 可用;Worker 的 client 設定裡**不存在**任何其他 team 的端點,這是硬隔離,不是紀律)。
- **作為 server(對外掛牌)**:整個 team 對外就是**一個 MCP server**;端點後面坐的是 Leader——所有進站呼叫先經過 §6.8 的收文管線,才會碰到部門內部。

### 6.6 呼叫方式:標準工具集(每個 team 都一樣)

Team MCP server 對外暴露兩組工具:

**A. 案件管理工具(業務無關,固定六個)**

| 工具 | 用途 | 輸入 → 輸出 |
|---|---|---|
| `describe_team` | 讀服務目錄:能力清單、狀態機、授權要求、目前負載 | (無)→ Agent Card |
| `delegate_task` | 提交委派 | 委派單(§6.2)→ `task_id` + 狀態 `Submitted` |
| `get_task_status` | 查案件進度 | `task_id` → 目前狀態 + 進度摘要 |
| `provide_input` | 回應「待補件」 | `task_id` + 補件內容 → 狀態回到 `Working` |
| `cancel_task` | 取消案件(觸發 §8 善後補償) | `task_id` → `Canceled` + 已執行的補償清單 |
| `get_task_result` | 取結案報告 | `task_id` → 結案報告(§6.4;artifacts 給引用) |

**B. 具名能力工具(由業務插槽自動生成)**

Team Charter 的每一項對外能力,自動生成一個具名工具(例如客服 team 的 `handle_complaint`、研究 team 的 `research_topic`)。參數結構就是委派單欄位(project/goal 照樣一級必填)——具名工具本質上是 `delegate_task` 的**預填能力版**,方便發件方的模型直接發現與選用。服務目錄與具名工具清單**永遠同步**(目錄改版,工具跟著改版)。

**產出物讀取**:結案報告裡的 artifacts 是**引用**;發件方按需以 MCP Resource 讀取(`artifact://team-id/project-id/...`),要多少拿多少——大檔案永不內嵌在訊息裡。

### 6.7 呼叫流程(端到端)

```
① 發現   Team A Leader 查部門名錄(registry)→ describe_team 讀 B 的服務目錄
② 授權   A 向名錄/授權中心取得對 B 的短效 scoped token(綁定能力範圍與期限)
③ 委派   A 呼叫 B 的具名能力工具(或 delegate_task),附完整委派單
         → B Leader 收文管線(格式審核→安全審查→調卷)→ 受理,回 task_id
④ 進行   短案件:同一連線內以事件串流回報進度直到結案(同步模式)
         長案件:先回 task_id,A 之後輪詢 get_task_status 或訂閱狀態通知(非同步模式)
         ——用哪種由委派單 constraints 的期限/預算決定
⑤ 補件   B 進入 Input-Required → A 收到通知 → provide_input 補件(或人工簽核後放行)
⑥ 結案   A 收到恰好一次結案通知 → get_task_result 取結案報告
         → 按需 read artifacts → 過自己的品管/DoD(§9)→ 採納並歸檔(§5)
```

**呼叫紀律**(對應各章制度):每次呼叫帶追蹤編號(§10 記帳)與防重複編號(§8);token 過期即斷(§7);A 對 B 的連續失敗觸發熔斷 → 查名錄換替代 team(§11);全流程狀態變更都落在雙方各自的存檔裡,事後可完整重播稽核(§8)。

### 6.8 Leader 的收發文管線

```
收文:格式審核(project/goal 必填)
  → 安全審查(來文一律視為不可信、過濾夾帶指令、確認在服務範圍內)
  → 調卷(以專案編號檢索自己檔案室)
  → 部門內規劃與派工(分工方式不外洩)
發文:彙整專員摘要 → 品管/驗收標準審查 → 結案報告
  → 歸檔(專案編號 + 來源標注)
  → 通知與記帳(追蹤編號傳遞、成功案件單位成本)
```

## 7. 部門規章與簽核(護欄與信任邊界)

1. **來文一律視為不可信**:其他部門的訊息 = 外部資料,過三道關卡(收文:夾帶指令與敏感資料過濾;執行:危險操作攔截與風險分級;發文:洩漏與合規檢查)。
2. **最小權限**:跨部門憑證用**短效授權**(分鐘級,案件結束即失效);對方只能委託服務目錄上有、且本次委派單允許範圍內的項目。
3. **委派深度與循環防護**:全鏈深度 ≤3;制度自動擋「A 委 B、B 又委回 A」。
4. **簽核點由制度設定,不由模型自報**:危險操作(業務插槽的清單)無條件觸發人工簽核;不依賴模型「覺得不確定時主動回報」。
5. **實證教訓**:「再檢查一次、務必完整」式的自動重做壓力,可能反而讓模型把夾帶的惡意內容照抄出來——**安全敏感的委派不疊加無差別的自我修正迴圈**。

---

## 8. 進度存檔與善後(檢查點與補償)

- 每個 team 自己存檔:工作每階段落一次進度,中斷可從斷點續作;歷史可回放稽核,也可開平行分支試不同做法。
- **跨部門的副作用必須可善後**:委派鏈上每個會改變外部世界的步驟,登記對應的「反向操作」;鏈上第 N 步失敗,依序倒退執行各部門的補償(如同專案中止時的違約善後條款)。
- 跨部門委託一律帶「防重複執行編號」:重試、重播都不會把同一件事做兩次(不重複扣款、不重複發文)。
- 長期案件靠存檔跨日續作;「待補件」暫停時封存進度、釋放人力。

---

## 9. 品質管理(驗證與回饋)

- **採納前必過獨立審查**:收到別部門的結案報告,先過自己的品管與當初委派單上的驗收標準,才併入自己的工作(信任但驗證)。
- **題庫制度(EDD)**:每個業務 team 上線前後,都用自己的驗收題庫回歸測驗(判定靠機械規則,不靠 AI 主觀評分,結果跨模型可重現)。
- **考制度,不考員工**:安全類題目驗收的是「危險操作**沒有被執行**」,而不是「模型沒有起念」——員工可以想,制度必須擋。
- 可逆的錯誤→讓當事人看到錯誤原因自我修正(有次數上限);不可逆的操作→強制簽核。

---

## 10. 成本與績效

- **追蹤編號跨部門傳遞**:每次委託的狀態、資源消耗、耗時、結果全程可查。
- **記帳單位 = 每成功案件成本**(不是每字成本):實證顯示便宜方案失敗重來反而總成本最高——「失敗很貴」。
- 模型分流是部門內建能力:複雜案件用高階模型、例行案件用低階模型,由制度自動路由。
- 三重硬上限(輪次/預算/深度)由制度執行;預算剩不到一成時強制存檔停工,等待指示。

---

## 11. 多 Team 協作(公司層)

- 協作形態限:**層級式**(總協調部門派往各部門)、**流水線**(部門間依序交棒)、**依賴圖**(按依賴關係並行),一律分階段驗收。
- **禁止全公司群聊式協作**(所有部門自由互聊):不可稽核、易陷入無限辯論、成本 15 倍。
- **備援 = 換部門**:某部門持續故障(熔斷器跳開)→ 按服務目錄找能力相符的替代部門;沒有替代→升級人工處理。
- 部門間對齊靠契約與驗收標準,不靠開會取得共識。

## 13. 運行基座(Runtime):Claude Agent SDK + 模型中立

### 13.1 執行引擎 = Claude Agent SDK

每個 agent(Leader、Worker、Evaluator)都以 **Claude Agent SDK** 作為執行引擎跑起來。規則書概念與 SDK 能力的對應:

| 規則書概念 | SDK 對應能力 |
|---|---|
| Leader 主迴圈(接案→規劃→派工→彙整) | SDK 主 agent(session) |
| Worker 專員(獨立工作空間、獨立工具授權) | SDK **subagents**——天然的 context 隔離與各自工具白名單,正合 §3 的派工制度 |
| 工具接入(§6 縱向流量) | SDK 原生 MCP client(每個 agent 掛自己授權清單內的 MCP servers) |
| Team 對外掛牌(§6.5 server 側) | 以 MCP server 包住整個 team 的 SDK 實例;端點後面是 Leader 的收發文管線 |
| 簽核與護欄(§7) | SDK hooks(工具執行前攔截 → 危險操作簽核點)+ 權限設定(允許/詢問/拒絕) |
| 進度存檔與續作(§8) | SDK session 續跑,搭配本規則書的檢查點與案件狀態機 |

**原則:SDK 是引擎,規則書是制度。** SDK 提供機械能力(迴圈、subagent、MCP、hooks、session);12 模組制度決定這些能力怎麼用。SDK 沒有涵蓋的制度——記憶主權與專案歸檔(§5)、跨 team 委派單/狀態機(§6)、跨部門善後補償(§8)、驗收題庫(§9)、每成功案件成本記帳(§10)——由 harness 層在 SDK 之上補足。

### 13.2 模型中立:所有模型都能用

執行引擎綁 SDK,**模型層不綁任何供應商**——這是沙漏後端接縫理念的落地:

1. **模型閘道(Gateway)接入**:SDK 對外講一種 API 格式;模型閘道負責把請求轉譯到實際供應商——原生 Anthropic(Claude)、**OpenAI 及任何 OpenAI 相容端點**、**Ollama 本機模型**、其他雲端供應商。換模型 = 改閘道設定,team 的制度與流程一字不動。
2. **逐角色指定模型**:業務插槽的「模型等級映射」(§4.1)直接對應到供應商與型號,且**每個角色可各自指定**——例如 Leader 用 Claude(規劃與對外溝通)、例行 Worker 用 Ollama 本機模型(省成本)、Evaluator 用 OpenAI(品管用不同家的模型,順帶降低「同一家模型自己審自己」的同源偏誤)。
3. **能力偵測與降級**:任何模型接入前先檢查最低能力門檻(至少:工具呼叫、結構化輸出);不達標的模型不承擔該角色,或降級到不需要該能力的職位。**制度設計不得依賴任何單一供應商的獨有功能**——所有 §5–§9 的制度都必須在「只會工具呼叫+結構化輸出」的模型上可運作。
4. **失效備援**:某供應商故障或限流 → 熔斷 → 按能力矩陣自動換下一家(§11「換部門」邏輯的模型版);全部失效才升級人工。
5. **成本分流的落地**:§10 的模型分流在此具體化——本機 Ollama 跑大量例行案件(零 API 成本),雲端強模型只出手複雜案件;記帳仍以「每成功案件成本」結算,避免「便宜模型失敗重來反而更貴」。

### 13.3 Team 定義檔包:agent 皆為 `.md` 檔 + skills + tools/scripts + hooks

一個 team 的全部定義**檔案化**成一個資料夾——「人事檔案、SOP 手冊、工具室鑰匙、規章執法器」全在裡面,即插即用、可版本控制、可整包交接。

#### 13.3.1 目錄結構

```
team-<名稱>/
  team.md                    # Team Charter:使命、能力清單 → 生成對外服務目錄
  agents/
    leader.md                # 每個 agent 一個 .md 定義檔(職務說明書)
    worker-<職能>.md          # 依 §4.2 四軸切分,一職能一檔
    evaluator.md
  skills/
    <技能名>/SKILL.md         # 業務 SOP 種子(§4.1 插槽),按需載入
  tools/
    mcp-servers.json         # 縱向工具接入設定(每個 agent 掛自己白名單內的)
    scripts/                 # 確定性腳本(制度的「手」)
  hooks/
    hooks.json               # 事件 → 腳本的對應設定
    *.sh / *.py              # hook 腳本本體
  golden-set/                # 驗收題庫(§9;機械判定規則)
  memory/                    # 部門檔案室(§5 命名空間,執行期生成)
```

#### 13.3.2 Agent 定義檔格式(`.md` = frontmatter + 職務說明書)

每個 agent 就是一份 Markdown:**frontmatter 定義「配置」,內文定義「職責」**。

```markdown
---
name: worker-<職能名>
description: <一句話說明何時該把工作派給它——Leader 據此選人>
model: <依 §13.2 逐角色指定:claude / openai / ollama 型號或等級>
tools: [<工具白名單——§4.2 四軸切分的結果,理想少於 10 項>]
skills: [<可載入的技能名>]
memory_scope: <可調閱的檔案室範圍(依 §5 命名空間)>
---

# 職務說明書(系統提示詞)

你是 <team 名> 的 <職能>。
- 職責:<具體做什麼>
- 硬性約束:<§3 對應約束——如 Worker:無對外窗口、完工只交結構化摘要、
  不得越過 Leader 向外委託;Leader:不親自執行業務工具>
- 工作方式:<簡報優先序、使用的表單格式、先寫理由再寫結論>
```

三個角色的定義檔重點:**leader.md** 的 tools 只有案件管理與委派能力(沒有任何業務工具);**worker-\*.md** 各自帶四軸切分出的最小工具白名單,description 寫清楚「什麼案子派給我」;**evaluator.md** 的 model 建議指定與執行者不同家(§13.2 降同源偏誤),tools 僅唯讀與驗收類。

#### 13.3.3 Skill 與 Tool/Script 的配置原則

三者分工,以「這一步需要什麼」判定:

| 需要的是 | 用什麼 | 形式 |
|---|---|---|
| **判斷力**(怎麼做、何時做、做到什麼程度) | Skill | `skills/<名>/SKILL.md`:名稱 + 觸發描述 + 步驟/準則;模型讀了才會——業務插槽的 SOP 種子就落地在這裡 |
| **外部能力**(碰得到系統與資料) | Tool | `mcp-servers.json` 掛 MCP servers;每個 agent 只掛自己 frontmatter 白名單內的 |
| **確定性**(每次都要一模一樣、不容模型發揮) | Script | `tools/scripts/`:格式驗證、防重複檢查、記帳、狀態機轉換——不經模型、可測試、可重複 |

判準口訣:**要判斷→skill;要能力→tool;要確定→script**。能寫成 script 的絕不交給模型(笨循環公理的落地)。

#### 13.3.4 Hook 配置:制度的自動執法點

Hook 是「制度不靠自覺」的機械落地——**模型可以忘,hook 不會忘**。每個 team 標配以下 hook(事件名對齊 Claude Agent SDK):

| Hook 事件 | 觸發時機 | 執行的制度 |
|---|---|---|
| **PreToolUse** | 任何工具執行前 | §7 簽核:比對危險操作清單→高風險擋下待人工簽核;檢查工具是否在該 agent 白名單內;檢查委派深度 ≤3 與循環委派 |
| **PostToolUse** | 工具執行後 | §10 記帳(token/耗時入帳)+ §8 稽核紀錄落檔(誰、何時、做了什麼、結果) |
| **SessionStart** | agent 開工時 | §3 工作簡報組裝(身份>工具>知識>進度>任務)+ §5 依 project 編號調卷注入 |
| **Stop** | agent 要收工前 | §6 結案報告格式強制(缺必填欄位→退回補齊)+ §9 DoD 自檢(沒過→不准收工) |
| **SubagentStop** | Worker 交件時 | §6.4 摘要格式檢查:超長、夾帶原始日誌、缺 status/confidence → 退回重寫 |

Hook 腳本一律是 script(確定性),設定收在 `hooks/hooks.json`;業務插槽的護欄政策(危險操作清單、風險分級)就編譯成 PreToolUse hook 的比對規則。

### 13.4 Team Gateway:把 team 資料夾掛牌成 MCP server

**一句話:team 資料夾是定義,Gateway 是掛牌機。** Gateway 是一支**業務中立的通用程式**——讀取任何 `team-<名稱>/` 資料夾,就把該 team 掛牌成一個 MCP server。每個 team 不寫自己的 server;同一支 Gateway 吃不同資料夾,就是不同部門。資料夾換了業務,server 的工具清單自動跟著變,制度層一行不改。

#### 13.4.1 啟動:把檔案「編譯」成 MCP 三原語

Gateway 啟動時讀定義檔包,生成 server 介面(全程確定性,無模型參與):

| 讀取 | 生成 |
|---|---|
| `team.md` 能力清單 | **Tools**:六個標準案件管理工具(§6.6A)+ 每項能力一個具名工具(§6.6B,參數=委派單欄位) |
| Team Charter | **Resources**:`agent-card`(服務目錄)+ `artifact://…`(產出物引用讀取) |
| 委派單範本 | **Prompts**:發件方可直接取用的委派範本 |

實作採官方 MCP SDK(TypeScript `@modelcontextprotocol/sdk` 或 Python FastMCP)。

#### 13.4.2 呼叫時:殼是手,裡面才是腦

工具被呼叫的處理順序,嚴格分兩層:

```
進站呼叫(delegate_task / 具名能力工具)
  → 【確定性層,Gateway script】收文管線前段:
     project/goal 必填檢查、能力範圍與白名單、token 驗證、防重複編號
  → 通過才【喚醒模型層】:以 Claude Agent SDK 啟動 Leader——
     agents 載入 agents/*.md、工具載入 tools/mcp-servers.json、hooks 載入 hooks/hooks.json;
     Leader 在 SDK 內以 subagents 派工給 Worker
  → 結案:Gateway 寫入案件庫、發出恰好一次結案通知
```

**狀態機由 Gateway 持有,不由模型持有**:`task_id`、案件狀態機(§6.3)、案件庫、防重複編號、記帳,全在確定性層。`get_task_status` / `get_task_result` 只是查庫——不喚醒任何模型,便宜且可靠。這正是腦手分離在 server 尺度的體現:**MCP server 的殼是手,SDK 裡的 Leader 才是腦。**

#### 13.4.3 Transport 與名錄

- **同機互連**:stdio——每個 team gateway 一個行程,對方 Leader 的 `mcp-servers.json` 直接掛它。
- **跨機互連**:Streamable HTTP——短效 token 掛在 HTTP 認證,狀態通知走事件串流;§6.7 的同步/非同步兩模式的自然載體。
- **名錄(registry)**:公司的部門名錄就是一份 MCP servers 清單(或本身也是一個小 MCP server,提供 `list_teams` / `find_team_by_capability`);新 team 掛牌=在名錄登錄端點。每個 Leader 的 client 設定只掛「名錄 + 被授權往來的 team」——Worker 依然一個橫向端點都沒有。

#### 13.4.4 快速驗證的捷徑

概念驗證階段,可用 `claude mcp serve` 把一個 Claude Code 實例直接變成 MCP server 充當 team——五分鐘能通。但正式架構必須回到 Gateway:狀態機、收文管線、記帳這些制度**必須在確定性層**,不能整包丟給模型。

## 14. 如何使用本規則書(實例化介面)

### 14.1 你要提供的輸入(業務模組設定表)

代入業務時,提供以下資訊(能填多少填多少,缺的由 AI 依 §4.2 推導並標注假設):

1. **業務是什麼**:一句話使命 + 這個部門對外承接哪些事
2. **工作單位**:一件「案子」長什麼樣(輸入是什麼、產出是什麼)
3. **要碰的系統與資料**:會用到哪些工具/系統/資料來源
4. **紅線**:哪些操作絕對危險、哪些需要你本人簽核
5. **品質標準**:怎樣算「做好了」(驗收怎麼判)
6. **量與預算**(可選):預期案量、成本敏感度、時效要求

### 14.2 AI 要交付的輸出(完整 team 定義檔包)

收到設定表後,依本規則書產出 **§13.3 格式的完整檔案包**(可直接放進 runtime 執行的資料夾),內含:

1. **`team.md`**:Team Charter + 服務目錄(名稱/使命/能力清單/對外窗口)
2. **`agents/*.md`**:Leader + 各 Worker + Evaluator 的定義檔(§13.3.2 格式)——frontmatter 配好工具白名單、模型、技能、記憶範圍;每個 Worker 的 description 附「四軸切分理由」(§4.2 Step 2/3)
3. **分工形態**:流程圖與階段驗收點(§4.2 Step 4,寫入 team.md)
4. **委派單與結案報告範本**(含 project/goal 欄位的業務化版本,作為 MCP Prompts)
5. **`skills/`**:業務 SOP 種子落地為 SKILL.md(§13.3.3)
6. **`tools/`**:mcp-servers 設定 + 確定性 scripts 清單(§13.3.3)
7. **`hooks/`**:五事件 hook 設定與腳本(§13.3.4)——護欄政策編譯進 PreToolUse
8. **記憶政策**:專案歸檔方式、各類記憶保存期(寫入 team.md)
9. **`golden-set/`**:10–30 題與機械判定規則
10. **成立流程確認單**(§4.3 逐項)

## 15. 附錄:設計法則 Checklist

**組織**
- [ ] 每 team 12 項制度自含;可獨立運作/成立/失敗/解散
- [ ] 跨 team 只走 Leader↔Leader over MCP;Worker 無對外窗口
- [ ] 內部分工對外不可見;外界只見服務目錄 + 案件狀態

**契約**
- [ ] 先定契約再開工;表單版本化,重大變更公告過渡
- [ ] project + goal 一級必填;自由散文式委託禁止
- [ ] 每個案件恰好一次結案通知

**記憶**
- [ ] 檔案室三級命名空間強制隔離;禁跨界查詢
- [ ] 溝通自含,不依賴對方記憶;專案編號 = 調卷與歸檔的鑰匙
- [ ] 外來資訊入檔:寫入驗證 + 來源標注 + 信任降級 + 限期保存

**安全**
- [ ] 來文一律不可信,過三道關卡
- [ ] 短效授權;能力範圍 = 服務目錄 ∩ 本次委派單
- [ ] 委派深度 ≤3;防循環委派;簽核點不交給模型自報
- [ ] 安全敏感委派不疊加無差別自我修正

**可靠**
- [ ] 跨部門委託帶防重複編號;副作用登記反向操作
- [ ] 輪次/預算/深度三重硬上限由制度執行
- [ ] 對方故障→熔斷→換部門→人工

**品質與成本**
- [ ] 採納前過獨立品管 + 驗收標準
- [ ] 每業務 team 有機械判定的驗收題庫
- [ ] 每成功案件成本記帳;追蹤編號全程傳遞

**業務實例化**
- [ ] 業務只出現在插槽;換業務不改制度
- [ ] 每個 agent 的存在都有四軸理由;說不出理由不拆
- [ ] 成立走標準流程:Charter→初始化→插槽→題庫→掛牌

**檔案化**
- [ ] 每個 agent 都是一份 `.md` 定義檔(frontmatter 配置 + 職務說明書),整包收在 team 資料夾
- [ ] Hook 覆蓋五事件:PreToolUse 簽核、PostToolUse 記帳稽核、SessionStart 簡報調卷、Stop 結案強制、SubagentStop 摘要把關
- [ ] 確定性步驟一律 script,不交給模型;要判斷→skill、要能力→tool、要確定→script
